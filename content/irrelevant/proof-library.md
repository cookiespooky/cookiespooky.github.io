# Библиотека доказательств

Позиционирование не должно держаться только на красивой формуле.

Если я говорю, что собираю из сложного материала рабочую карту для решений, это должно подтверждаться материалом: картами, схемами, разбором речи, примерами сайтов, пайплайнами, графами рекомендаций и результатами после работы.

Эта страница — место, где собираются будущие доказательства для сайта.

## Зачем нужна proof library

У сайта уже есть сильная внутренняя логика:

```text
речь / заметки / страницы / события
→ граф
→ карта
→ формулировка
→ решение / действие
```

Но для нового посетителя одной логики мало. Ему нужно быстро увидеть:

- как выглядит результат;
- что именно меняется после работы;
- какие артефакты он получит;
- где метод уже применялся;
- почему этому можно верить.

## Какие доказательства нужны

### Для LAT

Нужны обезличенные примеры:

- фрагмент речи до анализа;
- найденные языковые маркеры;
- карта роли и агентности;
- точка разрыва;
- новая формулировка позиции или действия.

Минимальный формат кейса:

```text
исходная формулировка → что было скрыто в речи → новая карта → следующий шаг
```

### Для позиционирования эксперта

Нужны примеры «до / после»:

- было: широкое самоописание;
- стало: категория, роль, отличие, короткая формула;
- как это развернулось в главную страницу, оффер или контент.

Хороший кейс должен показывать не только красивую фразу, но и выбор: от чего отказались, какую категорию заняли, с кем перестали конкурировать напрямую.

### Для Self Tracker

Нужны примеры карт по архиву заметок:

- повторяющиеся темы;
- петли;
- периоды действия и распада;
- сильные линии;
- темы, которые требуют решения;
- действие, которое меняет повторяющуюся структуру.

Важно не уходить в терапевтические обещания. Self Tracker должен доказывать наблюдаемость, а не обещать исцеление.

### Для Notepub

Нужны технические и визуальные доказательства:

- архив Markdown / Obsidian до публикации;
- структура будущего сайта;
- карта внутренних ссылок;
- примеры title / description;
- скорость и чистота HTML;
- опубликованный сайт;
- схема дальнейшего развития SEO-кластера.

Здесь доказательство должно быть максимально конкретным: был архив, стала публичная структура.

### Для Marketing OS

Нужны схемы пайплайна:

- вход: голосовое, заметка, тренд, идея;
- промежуточные статусы;
- артефакты: статья, пост, карусель, vault-запись;
- публикации;
- повторное использование контента.

Marketing OS нужно доказывать не «ИИ-магией», а управляемостью процесса.

### Для Recca

Нужны примеры графа рекомендаций:

- участники;
- события;
- роли;
- веса вклада;
- цепочка до сделки;
- где обычная рефералка увидела бы только один клик;
- что становится видно в графе.

Важно показывать Recca через коммуникации, события и вклад, а не через расплывчатые слова.

## Формат будущих кейсов

Каждый кейс лучше делать по одной структуре:

```text
1. Ситуация
2. Исходный материал
3. Что было не видно
4. Какая структура обнаружилась
5. Что было собрано
6. Какой следующий шаг стал возможен
7. Какие артефакты можно показать
```

## Что можно показать без раскрытия клиента

Не все кейсы можно публиковать полностью. Но почти всегда можно показать обезличенные артефакты:

- схему;
- карту;
- фрагмент формулировки;
- структуру сайта;
- до/после заголовков;
- граф без персональных данных;
- шаблон результата;
- набор критериев.

## Главная роль этой страницы

Эта страница фиксирует следующий слой развития сайта.

Первая итерация дала ясность позиции.  
Вторая итерация усиливает хабы.  
Следующий шаг — собрать доказательства, чтобы позиция стала не только понятной, но и проверяемой.
