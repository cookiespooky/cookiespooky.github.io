# Контент-пайплайн Marketing OS

Marketing OS нужна не для того, чтобы генерировать больше контента.

Она нужна, чтобы у каждого маркетингового артефакта был понятный маршрут.

```text
позиция → сырьё → черновик → визуал → ревью → публикация → архив → повторное использование
```

## Почему обычный контент-процесс ломается

В большинстве команд контент рождается из хаоса:

- голосовая в Telegram;
- заметка в Obsidian;
- транскрипт созвона;
- идея из встречи;
- список ключей;
- старый пост;
- бриф клиента;
- ссылка на конкурента;
- внезапная задача «сделать баннер к завтра».

Проблема не в том, что материала мало.

Проблема в том, что у материала нет статуса и следующего шага.

## Базовый workflow

Для большинства артефактов достаточно простой логики:

```text
draft → in_review → approved → published
```

Но сила Marketing OS в том, что этот workflow применяется не только к статьям.

Его можно использовать для:

- SEO-статей;
- постов;
- каруселей;
- рилсов;
- баннеров;
- презентаций;
- КП;
- тредов;
- афиш;
- продуктовых радаров;
- контент-планов.

## Входы

В систему могут попадать разные виды сырья:

```text
голосовые
транскрипты
заметки
брифы
темы
ключевые слова
старые статьи
посты
видео
созвоны
слайды
идеи из встреч
материалы из Obsidian
```

Важно не то, в каком формате пришёл материал.

Важно, чтобы он получил маршрут.

## Обработка

Marketing OS превращает сырьё в управляемые артефакты:

```text
голосовая → транскрипт → тезисы → статья / пост / сценарий
транскрипт → серия рилсов → карусель → FAQ
семантика → SEO-план → статья → публикация
позиция → message house → лендинг → посты → баннеры
статья → карусель → тред → короткие посты
слайды → презентация → видео → публикация
```

Каждый этап должен быть видим: кто создал, что получилось, какой статус, где лежит файл, что нужно сделать дальше.

## Роль стратегии

Без позиции пайплайн превращается в фабрику шума.

Поэтому Marketing OS должна начинаться не с кнопки «сгенерировать», а с карты проекта:

- категория;
- конкурентный контекст;
- verbal nail;
- against leader;
- лестница категории;
- reason to believe;
- do-not-say;
- visual hammer;
- продуктовые офферы;
- сегменты ЦА.

Тогда каждый новый материал работает не сам по себе, а на одну рыночную позицию.

## Human-review

Marketing OS не должна публиковать всё автоматически.

Правильный принцип:

> машина производит черновик, человек принимает решение.

Это защищает бренд от случайных обещаний, неверного тона, юридических рисков, плохих визуалов и стратегического распыления.

## Vault как память

После публикации материал не должен исчезать в ленте.

Он попадает в Markdown-vault и становится частью памяти проекта.

```text
пост → vault
статья → vault
баннер → vault
транскрипт → vault
решение по позиции → vault
proof point → vault
```

Так появляется накопление, а не вечная гонка за новым контентом.

## Почему это сильнее ручного контента

Ручной контент зависит от настроения, памяти и срочности.

Pipeline зависит от статуса.

Когда у каждого объекта есть статус, команда видит:

- что уже создано;
- что ожидает проверки;
- что готово к публикации;
- что опубликовано;
- где ошибка;
- что можно переиспользовать;
- какие темы не закрыты;
- какие артефакты связаны с каким проектом.

## Связь с Notepub

Notepub может превращать Markdown-граф в SEO-сайт.

Marketing OS может стать источником регулярного пополнения этого графа:

```text
Directus / vault → Notepub content repo → validate → build → deploy
```

Так сайт перестаёт быть разовым проектом и становится публичным слоем маркетинговой памяти.

## Главная формула

```text
не «сделать контент»
а «провести маркетинговый артефакт по маршруту»
```

Это и есть смысл Marketing OS.
